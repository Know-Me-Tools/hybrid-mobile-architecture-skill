This is an intentional machine handoff, not a failed implementation gate. The deployment
catalog, Dockerfiles, Compose profiles, Kubernetes overlays, CI workflows, documentation,
and skills are implemented. Native `linux/amd64` image completion and protected release
publication are deferred because compiling AMD64 under QEMU on the Apple Silicon workstation

## Remaining native build gate

Run these steps on an Intel/AMD Linux host with Docker Buildx. Start from a clean, current
`main`; do not use ignored artifacts copied from this workstation.

```bash
git pull --ff-only origin main
git status --short
docker buildx inspect --bootstrap

bash deploy/scripts/validate-catalog.sh
bash deploy/scripts/validate-gitops.sh
actionlint .github/workflows/*.yml

docker buildx bake -f deploy/docker-bake.hcl knowme-web \
  --set knowme-web.platform=linux/amd64 \
  --set knowme-web.output=type=docker

for target in flint-vault flint-meta flint-auth flint-hooks; do
  docker buildx bake -f deploy/docker-bake.hcl "$target" \
    --set "$target.platform=linux/amd64" \
    --set "$target.output=type=docker"
done

docker buildx bake -f deploy/docker-bake.hcl prometheus-postgres18 \
  --set prometheus-postgres18.platform=linux/amd64 \
  --set prometheus-postgres18.output=type=docker
```

Start PostgreSQL with development-only credentials, wait for readiness, then exercise its
extensions, logical replication, and WAL-G gate:

```bash
docker run -d --name prometheus-postgres18-verification \
  -e POSTGRES_USER=flint \
  -e POSTGRES_PASSWORD=development-only \
  -e POSTGRES_DB=flint \
  ghcr.io/prometheus-ags/prometheus-postgres18:dev

until docker exec prometheus-postgres18-verification pg_isready -U flint -d flint; do
  sleep 2
done

bash deploy/scripts/verify-postgres.sh prometheus-postgres18-verification
docker stop prometheus-postgres18-verification
docker rm prometheus-postgres18-verification
```

Then exercise all Compose profiles and render every Kubernetes option, including active F5
NGINX Ingress, NGINX Gateway Fabric, Traefik Ingress and Gateway API, Envoy Gateway,
cert-manager HTTP-01/DNS-01 wildcard certificates, and the external Certbot workflow.

## Release gate after native verification

Use the protected GitHub release workflow for the actual `linux/amd64` + `linux/arm64`
publication. It must publish the combined manifest, SBOM, provenance, signature, and scan
results; mirror the same digest to approved cloud registries; and open the external GitOps
promotion commit. Only that workflow's registry digests may replace
`PLACEHOLDER_RELEASE_DIGEST` in `deploy/images.lock.yaml`.

CI and GitOps ownership remains unchanged: CI builds, attests, mirrors, and proposes digest
promotion; ArgoCD owns live cluster state. Do not add `kubectl apply`, `helm upgrade`, or a
forced ArgoCD synchronization to the resume procedure.
