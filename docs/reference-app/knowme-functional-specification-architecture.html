01 · Overview

What KnowMe is.

KnowMe is a single application that ships on macOS, Windows, Linux, iOS, and Android from one Rust + React 19 codebase. It runs language models on the user's own hardware — Gemma, Qwen, Phi, and Mistral families — using mistral.rs on desktop (Metal) and llama.cpp on mobile, both behind the InferenceProvider trait (see versions.toml [inference]). It bundles eight thinking tools the user reaches for daily, six autonomous-agent templates that work without supervision, and a content-addressed plugin system that lets third parties extend the surface without ever touching user data.

Where most "AI assistants" are thin clients to someone else's server, KnowMe inverts the model. The local device is the primary runtime. The cloud is optional, opt-in, and exists only to sync with the user's own BossFang instances — which the user owns, runs, and shuts down at will. There is no central account that holds the user's conversations. There is no training pipeline that ingests their data. The conversation history, the agent memory, the encrypted skills registry — all of it lives in an embedded SurrealDB on the user's machine.

The result is an AI tool with the convenience profile of a consumer app and the trust profile of a piece of physical paper. It works on the subway. It works on a flight. It works in a SCIF. It works for the user the same way regardless of whether the user is online or whether KnowMe LLC is still in business.

02 · Philosophy

The four commitments.

KnowMe is the consumer surface for a particular set of architectural beliefs. These are not slogans. They are constraints applied to every decision in the codebase, from the choice of inference engine down to how filenames are cased.

Sovereignty is structural, not configurable.

Most "private AI" products express privacy as a setting. The user toggles a checkbox; the vendor promises the data didn't go anywhere. KnowMe rejects that model. Sovereignty is enforced by where the code runs, not by a checkbox in a settings panel. Inference happens on-device because the model weights and the user's prompt never leave the device — there is no "private mode" because there is no other mode. Cloud calls require explicit Cedar policy approval at the runtime level. The architecture is the guarantee.

Knowledge-Driven Development.

The user's own context — their files, their writing voice, their schedule, their preferences — is the model's most important input. KnowMe encodes that context into a local SurrealDB knowledge graph that grows with use. When the user asks a Hand to triage their inbox, the Hand reads from a KB the user owns, not from a model fine-tuned on a million strangers' inboxes. Reasoning is compiled at the user's edge, not resolved at someone else's query time.

Compile-time over query-time.

The expensive parts of agent reasoning — skill selection, tool routing, policy evaluation — are precomputed when the user activates a Hand or installs a skill, not on every invocation. The Cedar policy that says "this Hand may read mail but not send it" is a typed predicate at compile time, not a string parsed each time the Hand runs. The skill registry is a checked manifest, not a discovered set. The result is fast, predictable, auditable behavior on hardware that does not have to be a datacenter.

The user owns the substrate.

Every artifact KnowMe produces — Hands, conversation histories, skill registrations, model files, plugin manifests — lives in formats the user can export, version, and migrate. The SurrealDB file is on disk. The GGUF models are on disk. The plugin manifests are content-addressed CIDs the user can re-fetch from any IPFS gateway. If KnowMe LLC disappeared tomorrow, the user's KnowMe installation would keep working indefinitely. That is what sovereignty means in practice.

The contrast that matters

Closed assistants log every query, train on aggregate data, and cease to function the moment their vendor's billing system goes down. KnowMe runs on the user's hardware, learns from the user's own context, and continues to function regardless of upstream conditions. The trade-off is real: the most powerful frontier models are still cloud-only. KnowMe handles that by letting the user route specific calls to specific cloud providers — under explicit Cedar policy — rather than making cloud the default.

05 · Architecture

How KnowMe is built.

KnowMe is a Tauri 2 application. The frontend is React 19 with a strict layering discipline: components depend only on hooks, hooks subscribe only to stores, stores own all API calls. The backend is Rust, with platform-specific inference engines selected at compile time via Cargo cfg gates. State change propagates upward through subscription, never through callback drilling.

The seven layers in detail

Pure JSX. Receives props and hook returns. Never imports stores or APIs directly. Mobile and desktop differ only in layout components; the data flow is identical.

06 · Hands System

The three-step creation flow

Every Hand is created through a deterministic three-step modal. The constraint is intentional — predictability is what makes autonomous agents trustworthy.

①

Identity

Name, description, category. The Hand's "what it is and what it's for." Defaults seeded from the template.

②

Configuration

Model selection, schedule, skill grants. Per-skill consent — the Hand cannot use a skill not listed here.

③

Deployment

Run locally or on a connected BossFang instance. Cedar policy reviewed and accepted before activation.

Cedar policy enforcement

Every skill invocation by a Hand passes through Cedar at runtime. The policy is not a runtime instruction the model can be tricked into ignoring — it is a typed predicate evaluated by the Rust core after the model decides to call a tool but before the tool actually executes. A Hand whose policy says "may read mail but not send mail" cannot send mail even if a malicious email instructs it to. This is the structural defense against prompt injection that no LLM-layer mitigation can match.

First-use consent prompt

The first time a Hand attempts to invoke any skill, the user receives a consent prompt. The prompt shows the skill, the input, and the expected behavior. The user grants or denies. The grant is per-Hand, per-skill, and revocable from the Skills tab. Background drift is impossible — every new capability acquisition is a deliberate user action.

07 · BossFang Sync

Your devices talk to your servers.

KnowMe runs on the user's devices. BossFang (the production name for the librefang server) runs on the user's other infrastructure — a home Mac Pro, a rented VM, an enterprise cluster. The two communicate via OFP, the Open Fang Protocol, which is a peer-to-peer gossip layer over QUIC with encrypted application payloads. There is no central broker. There is no KnowMe LLC server in the path.

The OFP gossip protocol

OFP is what makes "your devices talk to your servers" not require a third party. Each instance maintains a peer list. State changes propagate through the peer graph using a Lamport-clock-based gossip with CRDT-friendly conflict resolution. Application payloads are encrypted with the user's identity key (a did:key: the user controls) and authenticated by mutual TLS. Without the user's key material, KnowMe LLC could not read sync traffic even if it sat on the wire.

No hub, no broker, no leak

OFP's design rule is that all four instance types are peers. The "Personal Cloud" instance is not privileged over the phone. The phone can be the source of truth for one entity and the home Mac Pro the source of truth for another. Conflict resolution is structural (CRDTs and Lamport clocks), not authoritative (server says so). This is what makes the architecture sovereign in the strict sense — there is no oracle.

08 · Plugin System

Extensibility without compromise.

Third-party developers extend KnowMe through WASM components built against the knowme:plugin@0.1.0 WIT contract. Plugins are content-addressed and distributed via IPFS. The host runtime is Wasmtime 41 with the Component Model enabled. Every plugin runs in a sandbox; every host capability is governed by Cedar; every UI surface is rendered in an isolated webview.

The WIT contract

The interface between host and plugin is defined in WebAssembly Interface Types — a typed IDL that produces Rust trait stubs at compile time via wasmtime::component::bindgen!(). The contract has two interfaces:

  • host — capabilities the runtime exposes to plugins in v0.1: log, storage-{get,set,delete}, uar-invoke, emit-event. Deliberately few, typed, and boring — the only successful consumer-app precedent for Rust→WASM extensions (Zed, Wasmtime + WIT in production since 2024) shipped a similarly restricted surface and expanded only with ecosystem pull.
  • plugin — lifecycle hooks the plugin must implement: init, handle-event, shutdown.

v0.2 candidates (not in the v0.1 contract)

Deferred until real plugins demonstrate the need, each with its own threat-model review (2026-07-16 assessment, rec #9): librefang-call (direct library surface), register-navigation (app-shell navigation injection), and the sandboxed-iframe plugin UI surface (window.knowme.invoke()/subscribe() bridge). Until then, plugins are headless capabilities reached through UAR skills and events.

The contract is versioned. Plugins compiled against @0.1.0 work against any host running 0.1.x for any x ≥ 0. Backwards-compatible additions bump patch. Because the contract is pre-1.0, a minor bump (0.2.0) signals a breaking change, per standard 0.x semver.

The plugin manifest

A plugin lives in IPFS as a directory: manifest.json, plugin.wasm, and (v0.2, once the iframe UI surface lands) ui-bundle.tar.gz. The manifest declares everything the host needs to install, sandbox, and govern the plugin.

FieldPurpose
idGlobally unique. Reverse-DNS recommended.
wasm.cid + sha256Content address of the WASM blob. SHA-256 verified after fetch.
ui.cid + entryv0.2 candidate — reserved; ignored by v0.1 hosts. Content address of the UI bundle (HTML / CSS / JS) and entry file.

Distribution and trust

Distribution uses IPFS for content addressing, with an HTTP gateway fallback for mobile (where running a local IPFS daemon is impractical). The fetcher tries the local daemon first, falls back to ipfs.io / cloudflare-ipfs.com / pinata.cloud, and verifies SHA-256 against the manifest after every fetch. Even a malicious gateway cannot serve a tampered binary — the hash check rejects it.

Trust is structural. Publishers sign their manifests with Ed25519 keys bound to a did:key: in the publisher record. Verified signatures display as a green pill in the plugin install sheet; unsigned plugins display as amber and require an explicit "trust unverified" toggle. The user always sees and approves the permission set before any code runs.

The sandbox

A plugin running in Wasmtime can call host functions for which it holds permission, allocate up to 64 MiB of linear memory, and run for up to 30 seconds per handle-event invocation (epoch interruption). It cannot open files outside its scoped storage, open network sockets directly, spawn threads, access the host's React tree, or call other plugins. Network access goes through uar-invoke("web-fetch", …) with a manifest-declared allow-list. Inter-plugin communication goes through the host event bus.

(v0.2 candidate — the plugin UI surface is not in the v0.1 contract.) When it lands, the plugin's UI bundle renders in an <iframe sandbox="allow-scripts allow-same-origin">. The host injects a small plugin-bridge.js that exposes window.knowme.invoke() and window.knowme.subscribe(), both of which translate to postMessage events routed to the plugin's WASM handle-event export. The plugin UI never touches the host's React state directly.

Why WASM Component Model and not Extism

Extism is the easier onboarding path — bytes in, bytes out, multi-language SDKs. KnowMe chose Wasmtime + Component Model because the surface includes UAR skill calls and BossFang HTTP — both involve structured data where loose typing would produce real bugs. The WIT file becomes a versioned contract enforced at compile time. The cost is steeper plugin-author onboarding; the benefit is a plugin ecosystem that does not silently rot as the host evolves.

09 · Privacy & Trust

What "sovereign" actually means.

"Privacy-first" appears in nearly every consumer AI product's marketing. KnowMe's claim is more specific and more verifiable: the application's privacy properties are structural consequences of where the code runs, not policy promises about what the vendor will do. If KnowMe LLC went bankrupt tomorrow, every user's installed copy would continue to function indefinitely with the same privacy properties.

The seven structural guarantees

GuaranteeHow it's enforcedFailure mode
Inference is on-device Model weights and prompts never leave the device. candle-vllm and llama.cpp run in-process. None — there is no network code path for local inference.
Conversation history is local Embedded SurrealDB. AES-256-GCM at rest. Device-key-derived. Disk forensics on an unlocked device. Mitigated by OS-level disk encryption.
No telemetry by default Telemetry disabled in default config. No event firing path absent the toggle. Crash reports remain disabled until explicitly enabled.
Cloud calls require Cedar approval Every outbound LLM call hits a Cedar policy check before request build. User explicitly authorizes cloud routing per provider.
Tool consent at first use UAR prompts the user for explicit grant on a Hand's first call to any new skill. User can pre-grant a skill's broad use in Settings.
Sync traffic is end-to-end encrypted OFP payloads encrypted with the user's did:key:. Mutual TLS authenticates peers. Compromised peer key. Mitigated by per-device key rotation.
Plugins are sandboxed Wasmtime memory + epoch limits. Cedar governs every host call. UI in isolated iframe. Bug in Wasmtime itself. Mitigated by version-pinning and security advisories.

On-device-only mode

The most aggressive privacy setting is "on-device-only mode." Toggling it rewrites the active Cedar policy to deny every cloud call, every outbound HTTP, every plugin network grant, and every BossFang sync. In this mode KnowMe is provably air-gapped — the user can verify by disconnecting the device's network and confirming nothing breaks. Some users (privacy-conscious professionals, regulated industries) live in this mode permanently.

What KnowMe does not do

  • No central account. KnowMe LLC has no database of users. The product works without registration.
  • No training on user data. User conversations and Hand outputs never reach any training pipeline. There is no pipeline.
  • No silent updates of permissions. A plugin's permission grant is immutable from install. Updating a plugin requires re-consent if the permission set changed.
  • No cross-tenant data flow. Each user's KnowMe install is an island. Sync goes only to peers the user has explicitly added.
  • No third-party analytics. No Segment, no Mixpanel, no Sentry-by-default. The user can opt in to crash reports if they choose.
The audit surface

Every consent decision, every skill invocation, every sync event, every plugin install is logged to a local audit trail the user can inspect from the Skills tab. The log is append-only and cryptographically chained — tampering is detectable. For users who need to demonstrate compliance to a third party (legal, medical, regulated industries), the audit trail is exportable as a signed bundle.

10 · Tiers & Access

Three tiers, one product.

KnowMe ships under a freemium model with two paid tiers above it. The free tier is genuinely useful, not a trial. The $20 tier is the consumer default. The $200 tier is the load-bearing tier — it is built around a powerful local model (Qwen-class) running on user hardware, which is what justifies the price without cloud API costs dominating the unit economics.

Why the $200 tier is load-bearing

Most "AI subscription" products at the $200 price point are reselling cloud API capacity at a markup. The unit economics are dominated by per-token API costs to Anthropic, OpenAI, or similar. KnowMe Pro inverts this: the value at $200 is a licensed local model running on the user's hardware, not a cloud reselling spread. The user pays for the licensing and packaging of the local Qwen build, the TurboQuant compression that makes it run on prosumer hardware, the air-gap support, and the priority plugin signing — none of which scale linearly with usage. This is what makes the tier sustainable at scale and explains why converting a $20 user to $200 is a higher-leverage activity than acquiring a new free user.

No tier ever takes away local sovereignty

The free tier and the $200 tier have the same privacy properties. Privacy is not a paywall feature — it is structural to the product. What scales with tier is access to bigger models, more Hands, more devices, and white-glove support. The architectural guarantees apply to every user.
