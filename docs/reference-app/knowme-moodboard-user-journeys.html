User moment · Morning open

"I just want to see what's running."

The user picks up the device. The first beat answers two questions: is the AI ready, and what can I do right now. The Engine Card resolves both — green dot, model name, throughput, RAM headroom — without making the user think about it.

  • Engine confirmation — "Ready" in green, with the active model and live tok/s. No hunting for status.
  • Capabilities at a glance — eight tiles, alphabetized by frequency of use, each with one-line subtitle and a colored pill that hints at the workflow class.
  • Sync awareness — bottom-left of the sidebar shows "3 instances · OFP" so the user knows their devices are talking.
Design intent

The home is an instrument panel, not a marketing page. Information density is high; visual noise is low. Pills and the ember accent (#FF6A3D) carry the only color — everything else is monochrome surface differentiation.

Screen 02 · AI Chat

Reasoning you can audit.

The chat surface streams tokens via Tauri channels — no polling, no spinner. Thinking mode is a per-message toggle: the assistant's reasoning trace renders in a collapsed cyan block that the user can expand. Tool calls appear inline. The conversation persists to the local SurrealDB and is full-text searchable from anywhere else in the app.

User moment · Mid-task question

"Show me the reasoning, not just the answer."

The user is mid-flow on something else — a task they need a quick decision on. They open Chat, ask a question, and want both the answer and a way to verify it. Thinking mode is the answer to that. It is collapsed by default, so casual users never see it. It expands on click for users who want to audit.

  • Streaming, not polling — tokens arrive over a Tauri channel as the model produces them. The first token shows up in 80–120 ms.
  • Thinking is auditable — the cyan reasoning block is the model's chain of thought, retained per-message, never silently discarded.
  • Model selector at top — Gemma for everyday, Qwen-35B for hard problems (routes through Home Mac Pro), Claude for cloud-class reasoning when policy permits.
Why this matters

Closed assistants present a finished answer with no inspectable reasoning. KnowMe's thinking mode is the structural opposite — every claim the model makes can be traced back to the steps it took. For users in regulated work, this is the difference between "useful" and "usable in a deposition."

Screen 03 · Hands

Agents that work for you while you sleep.

The distinction between Hands and Chat is the distinction between an agent the user works with and an agent that works for the user. The Hands tab is split: active Hands at the top — running, paused, or scheduled — and a templates strip below for spinning up new ones in three steps. Every Hand has a Cedar policy attached at creation; the policy travels with the Hand wherever it runs.

User moment · Reviewing morning digests

"What did my agents do overnight?"

The user opens Hands first thing in the morning. The Researcher digest from 9:14 AM is waiting; the Inbox Triage has run twice since they slept and surfaced 4 messages flagged "decision needed." The pattern repeats every day — and that consistency is what makes Hands feel like staff, not features.

  • Active section first — green status dot per Hand confirms it's running. Last-run time tells you when output was produced. Next-run tells you when to expect more.
  • Run count in ember — small but visible. The user sees their Hand has executed 128 times and built up real history.
  • "On Local" vs "On Home Mac" — the user knows which machine actually ran the work. A Hand that runs on the desktop's BossFang doesn't drain phone battery.
  • Template strip below — the same 6 archetypes from the spec, ready to deploy with one click into the 3-step wizard.
Design intent

Hands are not "automations." Automations are dumb cron jobs — Hands are agents with their own model, their own Cedar policy, and their own audit trail. The card layout shows enough metadata to convey that personality at a glance.

User moment · Spinning up a new agent

"I want this to feel like staffing a person, not configuring a script."

The wizard is short on purpose. Step 1 establishes who this Hand is: a name, a description in plain English, a category. Step 2 wires the brain — model, schedule, and the explicit list of skills the Hand may invoke (every grant is per-Hand and per-skill, never blanket). Step 3 picks where the Hand lives: this device, or any connected BossFang instance — with the Cedar policy visible in cyan before the user clicks Activate.

Why the wizard never grows

Every team that ships agent platforms is tempted to add "advanced options" to creation flows. KnowMe deliberately puts those into a separate Edit panel after activation. Creation is always Identity → Config → Deployment. New users can't get lost; experienced users can't slow themselves down by over-configuring upfront.

User moment · First-time sync setup

"My data goes only where I send it."

The user has just received a new device. They open Settings → Instances. They see exactly four rows — and they own three of them. The fourth, the team instance, was added when they joined Prometheus AGS, and the cyan "member" pill makes their role explicit: they participate in shared resources but don't administer them.

  • Status dot first — green means the peer is reachable now. A red dot would mean the peer is offline; the gossip log retries automatically.
  • Owner vs Member roles — ember "owner" pills for instances the user runs themselves; cyan "member" pills for instances they have access to but don't operate.
  • The URLs are the user's own — bossfang.local, bossfang.<you>.cloud. KnowMe's domain appears nowhere in this list.
What this enforces

"My data goes only where I send it" is a marketing claim from most vendors. Here, it's an architectural fact you can see and audit. The list of instances is the list of places your data goes — nothing routes through anywhere else.

User moment · Auditing what crosses the wire

"Show me exactly what is leaving this device."

The Sync card answers the question in one glance: 23 seconds since the last gossip round, 7 Hands tracked, 312 conversations replicated, 2.4 MB transferred. Below it, every entity type is a labeled toggle — the user can flip any of them off and the next OFP round will exclude that data from the gossip set.

  • Live status, not last-known — the green dot is bound to the OFP heartbeat. If a peer goes offline, the user sees it within seconds.
  • Models default off — a 4 GB GGUF transfer over a hotel Wi-Fi is the wrong default. The user enables it deliberately when at home.
  • Asymmetric exposure is opt-in — most categories are mutual, but skill registrations include credentials. The toggle off is a single click.
A different kind of trust

Most consumer products tell users "we don't see your data." KnowMe shows users exactly what they themselves are sending. Trust is not a paragraph in a privacy policy — it is a visible, toggleable, auditable surface.

User moment · Adding capability without compromising trust

"I want to extend KnowMe — but I want to see exactly what each plugin gets to do first."

A developer at the office shares a CID. The user pastes it into the installer. Within milliseconds the fetcher resolves the manifest from IPFS, verifies the Ed25519 signature against the publisher's did:key:, and the dialog renders: name, version, description, permission set, signature status. Nothing executes until the user clicks Install. The amber "unsigned" pill on dev plugins is impossible to miss; the green "Ed25519 verified" pill on the published plugin is exactly as visible.

The structural defense

Even a malicious gateway cannot serve a tampered binary — the SHA-256 in the manifest must match the bytes that arrived. Even an unsigned plugin cannot grant itself permissions — the manifest's permission set is reviewed by the user before WASM is instantiated. Plugin trust is structural, not contractual.

User moment · Picking the right tool

"Small for fast, big for hard, all under my control."

Gemma-3n is the daily driver — fast enough for chat, small enough to run alongside the editor. When the user needs frontier-class reasoning, they switch to Qwen-35B running on the Home Mac Pro. The 18 tok/s is slower than Gemma's 47 — but the model is 17 times bigger, and the call still happens against their own server, not a vendor's.

  • Active model marked clearly — green dot + ember "ACTIVE" pill. Other locals show as "USE" buttons; one click switches.
  • Acceleration backend visible — CUDA, Metal, Vulkan, CPU. The user knows which path their query is taking.
  • Remote models look different — cyan "Remote" pill makes it impossible to confuse with local. The amber "licensed" pill on Qwen-35B reflects the Pro tier inclusion.
  • Sideload GGUF — drop a fine-tuned checkpoint into the folder, KnowMe registers it. No platform gatekeeping.
A library, not an offering

Most consumer AI products show you their menu. KnowMe shows you your library. The distinction matters: a menu is what the vendor offers; a library is what the user owns. Sideloading is the proof — if you can drop in your own GGUF and run it, the platform isn't gating you.

How the surfaces compose into actual workdays.

Three journeys, one for each persona archetype. Each is a chain of moves across the eight tiles — the path the user actually walks, not a feature inventory. The screen references in ember link back to the mockups above.
