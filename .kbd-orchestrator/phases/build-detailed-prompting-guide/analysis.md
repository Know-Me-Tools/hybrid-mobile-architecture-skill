## Executive verdict

Retain the existing Docusaurus 3, local-search, and GitHub Pages stack. It already
builds, sanitizes, deploys, and serves the two current prompting pages. Replacing it
with Astro Starlight or another framework would not solve the actual deficit: the
missing ten recipes, six harness playbooks, executable loop examples, evidence-backed
model routing, case study, and verification contracts.

The correct split is:

- **Adopt as-is:** Docusaurus content plugins, autogenerated sidebars, existing local
  search, existing GitHub Pages Actions delivery, Agent Skills portability contract.
- **Adopt as validation dependencies:** Ajv, Playwright Test, and
  `@axe-core/playwright`.
- **Adapt:** Linkinator for bounded external-link checks and the existing
  `orchestrate-prometheus-application` skill for scenario/harness orchestration.
- **Reference:** the current OpenAI Proxy implementation as an evidence-bearing case
  study, without copying its undocumented authentication path or stale catalog prose.
- **Reject:** a second hand-maintained `site/docs/prompting` content tree and the stale
  `remark-lint-frontmatter-schema` package.
- **Build:** all Prometheus-specific content schemas, complete prompt recipes, harness
  instructions, registry-derived routing, case-study synthesis, and representative
  execution fixtures.

## Build-versus-adopt decisions by gap

### 1. Canonical source and publication

Use `docs/prompting` as the single authoritative content tree. Point the existing
prompting docs-plugin instance at `../docs/prompting`; the Docusaurus option contract
defines `path` relative to the site directory. Remove the two hand-authored site
summaries only after the external path passes a clean build. If a scratch build proves
that the repository layout or file watcher cannot consume the parent directory, the
fallback is a deterministic prebuild copy with a generated-file banner and parity hash.

Do not create another framework, another documentation repository, or two editable
copies. The existing multiple-plugin architecture already gives prompting independent
navigation and search ownership.

### 2. Scenario recipe corpus

No library can supply Prometheus-specific prompt packs. Build ten first-class Markdown
pages against one machine-checkable recipe contract. Each recipe must include:

- prerequisites and sources of truth;
- discovery and clarification prompt;
- Feynman learn/grade/gap prompt;
- KBD assess, analyze, spec, plan, apply, verify, and reflect prompts;
- bounded research and implementation prompts;
- producer and independent critic roles;
- authority, budget, retry, checkpoint, and termination rules;
- expected artifacts and observable public-boundary evidence;
- recovery prompt for failed or interrupted work;
- Karpathy project/private retention prompt;
- harness-specific invocation notes.

Ajv should validate recipe metadata and required sections. A project-owned Node script
must perform semantic checks that JSON Schema cannot: non-empty fenced prompts,
producer/critic separation, referenced skill existence, platform invariants, and a
declared public-boundary stop condition.

### 3. Harness playbooks

Build one page each for Codex, Claude Code, OpenCode, Kimi Code CLI, Google
Antigravity, and Zed. Agent Skills is the shared packaging baseline; it must not be used
to pretend the harnesses share commands, permission behavior, session ownership, or
headless modes.

Each playbook needs verified setup, instruction discovery, project skill paths, plugin
and MCP configuration, permission examples, plan/build/critic invocations, autonomous
budgeting, event/evidence capture, interruption/resume, and handoff. Copyable commands
must cite an official harness page and carry a `verified_at` date.

### 4. Feynman, KBD, Karpathy, PMPO, and autonomy loops

Build loop pages from the installed Prometheus skill contracts rather than paraphrased
memory. Show the actual command, prerequisite state, created files, closure rule, retry
limit, failure branch, human checkpoint, and next waypoint. Include a complete producer
to critic to retention transcript with explicit authority boundaries.

PMPO may improve metaprompts but may not modify requirements to make a failed result
pass. Feynman closure must use grade, transfer, and retention evidence. KBD examples
must use the canonical current-waypoint and handoff files, not stale aliases.

### 5. Model evidence and routing

Keep `model-registry.yaml`, but add a JSON Schema and evidence policy. Ajv validates
shape; project code checks unique IDs, exact source correspondence, date freshness,
reachable sources, non-empty claim support, and role vocabulary. Generate the public
routing tables from the registry so context windows, availability, and role guidance
are not duplicated in prose.

Source liveness is necessary but insufficient. A family announcement cannot alone
prove an exact API ID, a harness alias, or a claimed context window. Store confidence
and claim-to-source mapping, then fail publication when a required model has no current
official support.

### 6. OpenAI Proxy case study and skill-versus-agent decision

Reference the current source directly. The case study must distinguish:

- what the native-agent creator claims to generate;
- what the initial proxy skeleton contained;
- what the current repository actually implements;
- which capabilities were added later;
- what was verified through public routes/protocol consumers;
- what remains product-specific or unsupported.

The decision test is lifecycle-based: create a skill for repeatable instructions that
run inside a host; create a typed native agent when the capability owns a process,
protocol endpoint, durable state, concurrency, authentication, deployment, or an
independent release lifecycle.

Do not publish subscription-authentication implementation details as a supported vendor
contract unless current OpenAI documentation explicitly supports them.

### 7. Reusable orchestration skill

Adapt `orchestrate-prometheus-application` so it points to, selects, and validates the
new recipe/harness artifacts. Keep one template source and continue copying the
validated output to all six harness directories. The skill should remain compact under
the Agent Skills guidance; detailed examples belong in referenced docs rather than a
monolithic `SKILL.md`.

### 8. Verification and release

Adopt a layered gate:

1. Ajv and project semantic validators for registry, recipe, and harness contracts.
2. Docusaurus sanitization scanning the canonical root source.
3. Docusaurus production build with broken Markdown/internal links fatal.
4. Linkinator against the built site and bounded official external sources.
5. Playwright route/search/theme/responsive/copy tests.
6. axe scans plus keyboard and visual checks on representative routes.
7. Sitemap assertions for all required guide routes.
8. One full-hybrid recipe exercised in Codex and one ACP/CLI recipe exercised through
   OpenCode, Kimi, or Zed, with retained evidence.

The Pages workflow can run these gates before artifact upload. CI continues to publish
static output only; it must not expose private wiki or raw conversation material.

## Rejected alternatives

- **Switch to Astro Starlight:** rejected for this phase. The explicit goal and existing
  proven implementation are Docusaurus; migration adds risk without generating content.
- **Keep root summaries plus site summaries:** rejected because the current underdelivery
  is already hidden by this duplication.
- **Put all examples in the orchestration skill:** rejected because skills should remain
  focused and progressively disclose detailed references.
- **Generate prose directly from model output at build time:** rejected because public
  documentation must be reviewed, deterministic source.
- **Treat a live URL as proof of every model claim:** rejected; claim-level evidence and
  exact ID matching are required.
- **Use the frontmatter-schema remark plugin:** rejected because its registry release is
  stale and it validates a narrower surface than the required Ajv-based contracts.

## Risks and mitigations

| Risk | Impact | Mitigation |
|---|---|---|
| External docs path behaves differently in clean CI | Publication failure | Prove it with a scratch/clean build before deleting site summaries; keep deterministic-copy fallback. |
| Ten recipes drift structurally | Users receive incomplete prompts | Schema plus semantic validator and route inventory. |
| Harness commands age quickly | Broken or unsafe instructions | Per-command official citation and `verified_at`; bounded refresh job. |
| External link checks flake | Noisy CI | Timeout, retry, allowlist, and separate internal versus external policy. |
| Model routing becomes endorsement | Bad task/model selection | Date every entry, label vendor claims, show confidence, require local evaluation. |
| Case study overclaims generator output | False history | Compare initial/current commits and state inference explicitly. |
| Private memory leaks into Pages | Sensitive publication | Expand sanitizer to the canonical source and prohibit raw wiki imports. |
| Documentation becomes too large for skill context | Poor activation/use | Keep SKILL.md as router; progressively load only selected scenario/harness pages. |
