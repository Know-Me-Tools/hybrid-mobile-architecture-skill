## Why

The current guide reduces six materially different harnesses and multiple learning
and autonomy loops to short summaries. Users need verified, copyable invocations,
permission boundaries, evidence capture, recovery, and handoff behavior rather than a
fictional universal workflow.

> Scope: operational playbooks for six named harnesses and executable Feynman, KBD,
> Karpathy, PMPO, producer-critic, and autonomous-development loops.

## What Changes

- Publish one official-source, date-stamped playbook each for Codex, Claude Code,
  OpenCode, Kimi Code CLI, Google Antigravity, and Zed.
- Cover instruction discovery, project/global skills, plugins, MCP, permissions,
  plan/build/critic operation, autonomous budgets, event capture, interruption,
  resume, and handoff using the actual semantics of each harness.
- Publish executable Feynman, KBD, Karpathy, PMPO, producer-critic, cross-model,
  recovery, skill-generation, and native-agent-generation loop guides.
- For every loop, show prerequisites, exact invocation, created artifacts, authority
  envelope, retry limit, failure branch, human checkpoint, closure evidence, and next
  waypoint.
- Keep portable Agent Skills packaging as the shared baseline while explicitly
  preserving harness-specific differences.

## Capabilities

### New Capabilities

- `harness-playbooks`: Six verified operational playbooks with copyable setup,
  invocation, permission, evidence, and handoff examples.
- `learning-control-loops`: Executable learning, planning, implementation, critic,
  retention, and capability-generation loops with bounded termination behavior.

### Modified Capabilities

None.

## Impact

- Adds nested canonical pages under `docs/prompting/harnesses/` and
  `docs/prompting/loops/` plus their metadata/evidence records.
- Uses the content contracts from `prompting-guide-foundation` and the Agent Skills
  specification (`cand-007`).
- Does not change harness configuration or install tools; it documents verified use.

## Dependencies

- `prompting-guide-foundation` must define the metadata and evidence contracts first.
